AI SEO Agent Operating System

A practical overview of how Hermes, Telegram, per-client profiles, the VPS file system, and the Google Sheet work together to run SEO work with human approval.

Google Sheet = operating board Hermes = worker Telegram = conversation layer Profiles = client separation VPS = internal workspace
The simple mental model
Humans decide. Hermes works. The Sheet shows what matters.

The system should not mirror every message or expose every internal VPS file. It should show the user what needs attention, what is scheduled, what work is queued, what was approved, and what changed.

Daily operating rhythm

One daily refresh, plus live updates when work happens

1. Morning refresh
Hermes refreshes GSC data, cron status, approvals, tasks, and the Sheet. Mostly script-only, no strong model needed.
2. Nico checks Control Center
The top section shows approvals, blockers, broken jobs, and the next action. No need to inspect every tab unless something needs attention.
3. Scheduled jobs run
Cron jobs update the Schedule tab after they run. Failures rise to the Control Center. Routine success should stay quiet unless it creates something useful.
4. Work creates state changes
When Hermes starts, blocks, completes, or requests approval, it updates Agent Tasks, Approvals, and Activity Log. As-work-happens updates keep the Sheet accurate.
5. Approval gates action
Risky work waits for Nico to change approval status. Publishing, deploys, redirects, noindex/canonical changes, deletions, and outreach stay approval-gated.

Profiles and filesystem separation

Each client gets its own operating lane

Profiles prevent context and credentials from blending across clients. Workspaces keep the local data, reports, drafts, and site code organized on the VPS.

Approval loop

How a dropdown becomes action, safely

Today the Sheet is the control board. The next automation is a watcher that detects status changes and routes them safely.

StepWhat happens
1Hermes adds a row to Approvals when a task needs human approval.
2Nico changes Status from needs_review to approved, denied, or needs_changes.
3A watcher detects the changed row and logs the decision.
4Safe v1: it creates or updates an Agent Task and sends a Telegram summary.
5Later v2: approved, bounded actions can be executed automatically.
Recommended v1: dropdown approval creates a task and logs the decision. It should not auto-publish or deploy yet.

Review management for local SEO

Daily review checks without giving clients agent access

For local SEO clients, the agent can monitor reviews through Zernio or another GBP/review integration. The goal is simple: respond quickly to positive reviews and carefully route negative reviews for approval.

Review typeAgent actionApproval rule
Positive reviewDraft or post from an approved response pattern: acknowledge the review, thank the customer, mention the service naturally, and encourage them to come again or contact the business next time.Can use pre-approved templates once the client/agency approves the style.
Negative reviewDraft a calm response that acknowledges the review, thanks the person for the feedback, and asks them to contact the owner directly so the issue can be reviewed and rectified. Do not argue, blame, or admit legal fault.Must be approved before posting.
Unclear or risky reviewEscalate to the owner/agency with a short summary and recommended response angle.Must be approved before posting.
The tutorial can show the review workflow as a template even before a live GBP is connected. Zernio or another integration becomes the review source later.
